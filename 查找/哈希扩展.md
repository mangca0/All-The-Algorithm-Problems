## 在Java 中用 `security` 包实现常用的工程哈希算法
用`Security.getAlgorithms("MessageDigest")` 来获取支持的哈希算法：
`public static Set<String> getAlgorithms(String serviceName)`
```java
for (String algorithm : Security.getAlgorithms("MessageDigest")) {  
    System.out.println(algorithm);  
}
```

用`MessageDigest.getInstance(String algorithm)` 返回实现指定算法的 MessageDigest 对象：
`public static MessageDigest getInstance(String algorithm)throws NoSuchAlgorithmException`
```java
try {  
    md = MessageDigest.getInstance(algorithm);  
} catch (NoSuchAlgorithmException e) {  
    e.printStackTrace();  
}
```

用 MessageDigest 的实例方法 digest 完成哈希计算，返回存放哈希值结果的 byte 数组：
`public byte[] digest(byte[] input)`
```java
// 输入字符串返回哈希值
public String hashValue(String input) {
    // 用指定的字符集（UTF-8）把字符串转成字节数组
    byte[] hashInBytes = md.digest(input.getBytes(StandardCharsets.UTF_8));

    // 用 BigInteger 方便把二进制哈希值转成十六进制
    // 其中参数 1 表示这是一个正数（避免最高位为1时被当成负数）
    BigInteger bigInt = new BigInteger(1, hashInBytes);

    // 把这个大整数以 16 进制（十六进制）字符串输出
    String hashText = bigInt.toString(16);

    // 返回十六进制字符串
    return hashText;
}
```

## 哈希表原理，哈希表扩容
hashmap的增删改查平均O(1)，按数据的哈希值取模的结果，把数据均匀分散在不同的桶中
扩容是**全量扩容**
不同语言桶的结构不同，但都只影响常数时间复杂度，平均O(1)是不变的
- Java HashMap 为链表桶，当桶规模达到阈值会树化为红黑树
- C++ unordered_map 链表桶
- Python 采用开放寻址
- Go 开放寻址，桶满时使用溢出桶

## 布隆过滤器
本质是一个很长的位数组和一系列随机哈希函数
**常用于检索一个元素是否在一个集合中**，常作为黑名单系统，去重器，数据定位
### 算法原理
当一个元素被加入集合时，通过 k 个哈希函数将这个元素映射成一个位数组中的 k 个点，把它们置为1
检索时，把元素处理为 k 个点，如果这些点有任何一个0，则被检元素一定不在；如果都是1，则被检元素很可能在
远远超过哈希表实现的时空效率，但无法避免的是有一定的误差：可能会产生假阳性（误报），但不会产生假阴性（漏报）
### 参数值
位数组长度 m，与误差率 p 大致成反比例函数图像。m 越大，p 越低，相应的开销也就越大，一般通过锚定 p 来确定 m 的取值
哈希函数个数 k，与误差率 p 大致成二次函数图像，k 过小或过大，都会导致 p 过高
实际上，m 和 k 有一套可计算的理论最佳值：
$m = -\frac{nlnp}{(ln2)^2}$  $k = ln2\frac{m}{n}$
根据 m，k 实际情况倒推出的 p：$p_{实际} = (1-e^{-\frac{nk}{m}})^k$

## 一致性哈希
在一致性哈希下，桶个数的增删平均只需要对 m/n 个键值对重新映射，其中 m 是总键值对的数量，n 是桶数量。然而在传统的哈希表中，添加或删除一个槽位的几乎需要对所有键值对进行重新映射
### 算法原理
下面用「服务器」和「数据」来实例化「桶」和「键值对」
#### 数据插入
一致性哈希旨在避免在集群中添加或移除服务器时需要重新分配每个数据的问题，其核心思想是**使用一个哈希函数，将数据对和服务器映射到一个单位圆上**（定义区间的哈希函数不一定和计算服务器哈希值的函数相同，但是两个函数的返回值的范围需要匹配）
每个数据对会按顺时针方向分配给圆上出现的下一个服务器。通常使用二分查找或线性查找分别以 O(log⁡N) 或 O(N) 的复杂度找到放置的位置
#### 增删服务器
当添加或移除服务器时，绝大多数数据对会保留其先前的服务器分配，将所有映射属于该服务器（即哈希值小于新服务器哈希值，从图上就是逆时针寻找，直到找到一个另一个服务器的区域）的数据进行迁移
在实践中，常使用二叉搜索树 (BST) 来维护哈希环中的服务器，使用 BST 来查找服务器的后继或最小值
### 虚拟结点技术
一般一致性哈希带来以下问题：
- 服务器数量少，哈希所在位置不均匀，**数据倾斜**
- 要人为均匀分配服务器位置吗？一旦增删服务器就不均匀了

解决：**虚拟结点技术**
给真实服务器分配多个“子服务器”，区别在于原来每个服务器对应一个节点，现在每个数据库会对应若干个节点
不再自己瓜分区域，而是子服务器瓜分

这一技术能解决**数据倾斜**的问题，在实际场景中，还能有效处理服务器故障，或单个数据被大量访问的问题
还能实现**负载管理**，根据服务器处理能力分配不同数量的节点数
